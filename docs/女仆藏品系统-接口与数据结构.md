# 女仆藏品系统接口与数据结构

## 1. 文档目标
本文档用于定义女仆藏品系统在第一阶段上线所需的数据结构、接口协议、业务校验和错误返回，供前后端联调使用。

## 2. 枚举定义

```ts
type Rarity = 'R' | 'SR' | 'SSR'
type UnlockType = 'summon' | 'stage' | 'event'
type LoveStage = 'stranger' | 'familiar' | 'close' | 'love' | 'oath'
type StoryUnlockType = 'level' | 'loveStage' | 'levelAndLoveStage'
type GiftTag = 'dessert' | 'flower' | 'music' | 'tea' | 'luxury'
```

## 3. 静态配置表设计
静态配置建议由配置表、CMS 或 JSON 文件管理，由前端在启动时预加载，后端只持久化玩家进度。

## 6. 数值与业务规则

### 6.1 升级规则
* 初始等级上限为 20，突破后扩展到 40、60、80。
* 单级所需经验建议先使用 `100 + level x 20`。
* 加成计算公式：`当前加成 = 基础加成 + (等级 - 1) x 成长加成 + 突破附加值`。
* 突破节点为 20、40、60 级。

### 6.2 好感规则
* 0 到 99 为陌生。
* 100 到 299 为熟悉。
* 300 到 699 为亲密。
* 700 到 1199 为恋慕。
* 1200 以上为誓约。
* 普通礼物增加 10 点好感。
* 偏好礼物增加 20 点好感。
* 讨厌礼物增加 5 点好感。
* 每日首次互动增加 5 点好感。

### 6.3 故事解锁规则
* 第一章：女仆解锁后直接开放。
* 第二章：达到 10 级开放。
* 第三章：达到 20 级且好感达到熟悉开放。
* 第四章：达到 40 级且好感达到亲密开放。
* 第五章：达到誓约阶段开放。

## 8. 服务端校验规则
* 未解锁女仆不能进入升级、送礼和互动接口。
* 队伍中不能出现重复女仆。
* 看板位只能设置为已解锁女仆，也可以传 `null` 表示取消。
* 升级时必须校验经验道具库存是否充足。
* 当前等级达到上限但未突破时，升级接口应直接返回错误。
* 礼物数量必须大于 0，且必须校验背包剩余数。
* 每日互动奖励每天只生效一次，多次点击只返回互动结果，不重复增加好感。
* 故事章节是否解锁由服务端统一计算，前端只根据返回状态展示。

## 9. 错误码建议

```json
[
  { "code": "MAID_NOT_UNLOCKED", "message": "女仆未解锁" },
  { "code": "ITEM_NOT_ENOUGH", "message": "道具数量不足" },
  { "code": "LEVEL_CAP_REACHED", "message": "已达到当前等级上限" },
  { "code": "INVALID_TEAM_SLOT", "message": "上阵配置不合法" },
  { "code": "SHOWCASE_INVALID", "message": "看板角色不可用" },
  { "code": "INVALID_PARAM", "message": "请求参数错误" }
]
```

## 10. 前后端职责边界
* **前端负责**：界面状态切换、抽屉拖拽、动画反馈、筛选排序、详情页签切换、接口成功后的局部刷新。
* **后端负责**：升级结算、好感结算、故事解锁判断、库存扣减、队伍合法性校验、完整状态返回。
* **配置侧负责**：女仆基础资料、技能文案、故事章节文本、礼物标签、解锁条件。

## 11. 联调建议
* 第一轮联调优先打通 `collection`、`detail`、`upgrade`、`gift` 四个接口。
* 前端在没有真实接口前，先按本文档的返回结构建立 mock 数据。
* 写接口后统一由后端返回最新女仆状态，前端不要在本地自行推算好感阶段和故事解锁状态。
